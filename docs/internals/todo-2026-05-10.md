## 1. Duplicated marshalling â€” the worst offender

There are effectively **four parallel Pythonâ†”C conversion trees** that should be one:

| Path | Files | Direction |
|---|---|---|
| Generic GIArgument | `runtime/marshal.c` (`pygir_argument_from_py`, `_to_py`) | both |
| Per-call invoke | `runtime/marshal.c::_for_call` + `invoke/bind.c` + `invoke/return.c` | both |
| Callback trampoline | `runtime/closure.c` (`storage_tag_value_to_py`, `c_arg_to_py`, `c_array_to_py_list`) | both, **forked** |
| GValue pack/unpack | `runtime/gvalue.c` | both, off the GType axis instead of GITypeTag |
| JIT scalar helpers | `src/jit/helpers.c` (40+ tiny `h_int8_to_hpy` etc.) | both |

The **callback path in `closure.c`** is the most concerning: it dispatches scalars off `ffi_type*` rather than `GITypeTag`, so it has independently re-derived everything in `scalar.c`. Same enum/int conversion implemented 3â€“4 times. `c_array_to_py_list` (closure.c:458) re-implements the array element loop already in `runtime/c-array.c`.

`runtime/effective.c` / `effective_bind.c` are **not** a parallel marshaller â€” they're metadata/lookup helpers exposed to tests. Confusing names, but no duplication there.

**Right shape:** one tag-driven dispatcher (the marshal.c one), called from invoke, closure, GValue, and field access. Closure should not be a fork.

## 7. Memory-safety story

What exists:
- `make valgrind` with proper suppressions (Python, GLib, GTK, pygobject) â€” runs full pytest under memcheck, error exit code on definite leaks.
- `make test-debug` runs against debug-build CPython with `MALLOC_CHECK_=3 MALLOC_PERTURB_=45`.
- `g_autoptr(GIBaseInfo)` cleanup is used.
- Toggle-refs explicitly avoided (replaced with `g_object_weak_ref`).
- `__grefcount__` / `__gpointer__` debug accessors landed in `08dae92`.

What's missing:
- **No ASan / UBSan / LSan build flavor.** No meson option exposes `b_sanitize`. This is a 1-line addition that would catch a large class of issues `make valgrind` misses (UB, stack-use-after-return, container-overflow).
- **No CI leak target.** `make valgrind` is manual.
- **No per-test isolation under valgrind** (it's an aggregate run).
- **No refcount-stress** (millions of allocs + sys.getrefcount drift). PyGObject historically leaked here.
- **No GIBaseInfo lifecycle audit.** The plan doc itself notes "1.2% of hot loop in `gi_base_info_unref` alone" â€” that's a hint that the refcount traffic deserves a directed test.
- **HPy compat layer** has no leak mode.

Concrete suggestion: add `-Db_sanitize=address,undefined` build dir, wire `make test-asan`, gate it on debug Python with `_PYTHON_MALLOC_=malloc`. Single highest-value addition.

## 8. C vs Python â€” what does "force C" actually buy us?

The honest accounting:

- **Cold paths (lazy resolution, descriptor build, plan compile, namespace open, override registration):** ~1.5â€“2 KLoC in C that runs *once* per symbol. The wall-clock benefit of writing this in C vs Python is small in absolute terms (these run microseconds-per-symbol once), but the wins compound because the cached descriptor is then used millions of times. The real value isn't speed â€” it's that **descriptor caching becomes a static, debuggable structure** instead of a dict of half-built closures. Worth keeping in C.
- **Hot path (call dispatch, return shape):** ~600â€“800 LOC of C plus the JIT trampolines. Here C buys 15â€“25% on common method calls, but the dominant cost is **unavoidable Python-side cost** â€” `PyUnicode_AsUTF8` per string arg, refcount bumps, type marshaling that any binding has to do. A hypothetical pure-Python implementation using the same plan structures would likely run within 2Ã— of goi on string-heavy calls and within 3â€“4Ã— on int-only calls.
- **JIT (`src/jit/`)** beats libffi on int/pointer-heavy signatures by ~3Ã— call overhead by emitting native trampolines that skip per-call ffi_cif setup. `PYGIR_CALL_MODE=AUTO` defaults to JIT with FFI fallback. That's where forcing native code actually pays off.
- **Python still in the per-call path?** No. Once a descriptor is built, the interpreter goes vectorcallâ†’C. `Py_TPFLAGS_METHOD_DESCRIPTOR` avoids `PyMethod` allocation. So the existing Python in `pygobject/gi/` sits in the cold path only.

**Bottom line on "force C even in cold path":** The benefit is *correctness and structural clarity*, not throughput. Keep cold paths in C where they integrate with descriptor caches, but stop writing C just to feel fast â€” the override migration to TOML+codegen is a better lever than rewriting `_signalhelper.py`.
