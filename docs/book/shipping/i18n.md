# Internationalization (gettext)

> Translation infrastructure that lives in your repo so it can be shared across distribution targets. Cross-references the [GNOME localization chapter](../gnome/localization.md) which covers GNOME-specific workflow.

## What this chapter covers

- Why this chapter is in Shipping: i18n affects every target (desktop file, metainfo, schemas, app strings).
- The gettext model recap: `.pot` → `.po` → `.mo`.
- Marking strings in:
    - Python (`_`, `N_`, `ngettext`).
    - `.ui` and `.blp` (`translatable="yes"`).
    - `.desktop.in`, `.metainfo.xml.in`, `.gschema.xml.in`.
- `POTFILES.in` enumeration.
- Meson `i18n` module: extract, update, compile, install.
- Domain setup at runtime: `bindtextdomain`, `textdomain`, locale fallback chain.
- Translator workflow:
    - Self-hosted (`po/` in your repo, accept PRs).
    - Weblate.
    - GNOME Translation Project (Damned Lies) — for GNOME-platform apps.
- Plural forms — get them right with `ngettext`.
- Right-to-left and locale-aware formatting (`Glib.format_size`, `Glib.DateTime`).
- Pseudo-localization for testing.
- Per-platform locale quirks (Windows env, macOS preferences, Flatpak runtime locale).

## What you'll be able to do

- Set up gettext from scratch.
- Ship translated strings and metadata.
- Accept translation contributions cleanly.
